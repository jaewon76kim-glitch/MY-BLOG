## 왜 이 주제인가

위성통신 교재를 읽다 보면 도플러 천이와 전파지연(RTT)이 항상 수식으로만 등장한다. 위성이 지평선에서 떠올라 천정을 지나 다시 지평선으로 지는 동안, 이 두 값이 시간에 따라 어떻게 변하는지 눈으로 직접 보고 싶어서 `apps/doppler-sim`을 만들었다. LEO(저궤도)부터 GEO(정지궤도)까지 궤도 고도를 바꿔가며 비교할 수 있는 인터랙티브 시뮬레이터다.

## 핵심 물리

### 도플러 천이

위성과 지상국 사이의 상대 속도 성분이 캐리어 주파수를 밀거나 당긴다.

```
f_d = (v / c) * cos(θ) * f_carrier
```

- `v`: 위성의 궤도 속도
- `θ`: 위성 진행 방향과 시선(line-of-sight) 사이의 각도
- `f_carrier`: 캐리어 주파수

위성이 지상국을 향해 접근할 때는 양(+)의 천이, 천정을 통과하는 순간은 상대 속도의 시선 성분이 0이 되어 천이가 0을 지나고, 멀어지기 시작하면 음(-)의 천이로 바뀐다. 그래프에서 이 0 교차 지점이 정확히 통과 중간 지점과 겹치는 걸 확인하는 게 이 앱의 재미있는 포인트였다.

### 왕복 전파지연(RTT)

경사거리(slant range) `R`을 앙각(elevation) `El`의 함수로 근사해서 구한다.

```
R  = √(h² + R_E² − 2h·R_E·sin(El)) − R_E
RTT = 2R / c
```

- `h`: 위성 궤도 고도
- `R_E`: 지구 반지름
- `El`: 현재 앙각

앙각이 최대(천정에 가까울수록)일 때 경사거리가 최소가 되고, 따라서 RTT도 최솟값을 찍는다. LEO(550km)는 RTT가 수 ms 단위인 반면 GEO(35,786km)는 편도만 120ms 안팎이라 왕복이면 240ms를 넘는다 — 화상통화에서 느껴지는 지연이 왜 위성 링크마다 이렇게 다른지 숫자로 체감할 수 있다.

## 구현

파일 구조는 블로그 나머지 부분과 완전히 분리했다.

```
/apps/doppler-sim/
├── index.html
├── css/style.css
└── js/
    ├── sim.js       # 궤도·도플러·RTT 계산
    ├── renderer.js  # Canvas 위성 애니메이션
    └── chart.js     # Chart.js 래퍼
```

- **Canvas 애니메이션**: `requestAnimationFrame`으로 위성이 지평선→천정→지평선을 호(arc) 형태로 이동하는 모습을 그린다. 3D 지구본 대신 2D 단면도로 단순화했는데, 오히려 앙각 변화가 더 직관적으로 보였다.
- **Chart.js 실시간 그래프**: 도플러 천이와 RTT를 각각 시간축 라인 차트로 그린다. CDN 대신 로컬에 `chart.umd.min.js`를 복사해 넣어서, 오프라인에서도 앱이 그대로 동작한다.
- **파라미터 슬라이더**: 궤도 고도(400~35,786km), 캐리어 주파수(1~30GHz), 최대 앙각(10~90°)을 조절하면 애니메이션과 그래프가 즉시 다시 계산된다. 값을 바꿔가며 "LEO는 통과가 몇 분 만에 끝나는데 GEO는 왜 거의 정지해 보이는지"를 바로 비교할 수 있다.

## 이번 버전에서 뺀 것

범위를 명확히 하려고 처음부터 제외한 것들이다.

- 3D 지구 구체 렌더링(Three.js 등) — 2D 호 표현으로 대체
- 실제 TLE(Two-Line Elements) 파싱과 실시간 위성 추적 — 원형 궤도 근사만 사용
- 다중 위성 동시 표시(Starlink 군집 시뮬레이션)
- 대기 굴절, 전리층 보정, 강우감쇠 등 링크버짓 계산

핵심 개념(도플러 0 교차, RTT 최솟값 지점)을 전달하는 데 필요한 만큼만 구현하는 게 목표였다.

## 배운 점

수식을 눈으로 확인 가능한 그래프로 옮기고 나니, 교재에서 "천정 통과 시 도플러 천이는 0"이라고 읽었을 때보다 슬라이더로 고도를 바꿔가며 그 0 교차 지점이 이동하는 걸 직접 볼 때 훨씬 오래 기억에 남았다. 앞으로도 개념을 다룰 때는 정적인 설명보다 조작 가능한 시뮬레이터로 만드는 쪽을 우선해볼 생각이다.

# MY-BLOG 글 올리기 워크플로우

CLAUDE.md의 "작업 사이클"은 원래 웹앱 주제(apps/ 폴더, doc/ 하위 pdf·tex 기반)를 위한 것이지만,
사용자는 텍스트 글(포스트)을 올릴 때도 같은 규율 — **승인 없이 구현을 시작하지 않는다** —
을 지키길 원한다. 이 스킬은 그 규율을 글쓰기 작업에 맞게 적용한 버전이다.

## 언제 이 스킬을 쓰는가

- "MY-BLOG에 글 올려줘", "포스트 하나 써줘" 같은 명시적 요청
- 주제가 이미 있는 웹앱(apps/*)의 개발 후기·회고를 글로 남겨달라는 요청
- CLAUDE.md를 언급하지 않아도, MY-BLOG 프로젝트 디렉토리에서 "글/포스트/article"을
  작성·게시해달라는 요청이면 항상 적용

적용하지 않는 경우: 이미 있는 글의 오탈자 수정처럼 사소한 편집, 또는 사용자가
명시적으로 "계획 없이 바로 써줘"라고 승인 절차 생략을 요청한 경우(그래도 결과를
보여주고 확인은 받는다).

## 절차

### 1단계: Plan (승인 전까지 파일을 쓰지 않는다)

- **글 주제도 웹앱과 마찬가지로 항상 `C:\Users\hugok\Claude\Projects\doc\`
  (WSL: `/mnt/c/Users/hugok/Claude/Projects/doc/`) 하위 폴더 중 하나의 내용을 기반으로
  정한다.** 블로그 운영기(예: doppler-sim 개발 후기)처럼 프로젝트 자체가 소재인 글이
  아니라면, "일반 기술/회고 글"이라 해도 소재 없이 임의로 주제를 지어내지 않는다 —
  doc 폴더 목록을 사용자에게 제시하고 어느 하위 폴더 내용을 다룰지 먼저 정한다.
  - doc 폴더 하위 각 항목의 성격(PDF 교재 프로젝트 vs 요약 프로젝트 vs 기타 개인
    프로젝트)은 `doc/CLAUDE.md`에 정리되어 있으니 먼저 읽는다.
  - 소스 파일은 `.tex`가 있으면 `.tex`를 우선 사용한다([[feedback_use_tex_not_pdf]]).
  - 이미 다룬 폴더(예: 위성통신 → doppler-sim)라도, 같은 폴더의 다른 절/주제를 다시
    다루는 것은 허용된다. 다만 겹치는 내용이 없도록 이전 글을 확인한다.
- 주제가 없으면 사용자에게 묻는다. 최근 커밋/spec.md/review.md 등 프로젝트 맥락에서
  자연스러운 후보(예: 방금 만든 웹앱의 개발 후기)가 있으면 그것을 먼저 제안한다.
- 아래 항목을 정리해 사용자에게 제시하고 **승인을 받는다**:
  - 슬러그(slug, 영문 kebab-case, `posts/index.json`의 다른 slug와 중복되지 않게)
  - 제목, 날짜(YYYY-MM-DD, 오늘 날짜 기본), 태그, 한 줄 설명(description)
  - 본문 구성(outline) — 섹션 제목 목록 정도의 개요
- 웹앱 개발 후기라면 해당 앱의 `spec.md`/`review.md`를 읽어 내용 근거로 삼는다.
- 계획은 spec.md 파일로 따로 저장할 필요 없이 대화 중 제시 + AskUserQuestion으로
  승인받는 것으로 충분하다(글 하나짜리 작업에 파일까지 만드는 건 과함).

### 2단계: Build

- `posts/{slug}.md` 파일을 만든다. 기존 포스트(`posts/hello-world.md`,
  `posts/today-i-learned.md`, `posts/markdown-guide.md`)와 동일한 frontmatter 형식을 따른다:
  ```
  ---
  title: ...
  date: YYYY-MM-DD
  tags: [태그1, 태그2]
  description: ...
  ---
  ```
- 본문은 `##` 레벨 섹션으로 나누고, 기존 글들의 톤(1인칭 회고/설명체, 코드블록과
  표 적절히 사용)을 따른다.
- `posts/index.json` 배열 맨 앞에 새 항목을 추가한다(slug, title, date, tags,
  description, wordCount). `wordCount`는 frontmatter를 제외한 본문을 공백 기준으로
  분리한 단어 수(대략치, 정확할 필요 없음 — `readingTime` 계산에만 쓰인다).
- CLAUDE.md의 "웹앱 규칙"에 나오는 `/apps/{앱이름}/` 임베드(index.html 카드 추가)는
  **글 작업에는 해당하지 않는다** — 글은 `posts/index.json`에 등록되는 것 자체가
  게시(publish)다. index.html이나 다른 페이지는 건드리지 않는다.
- 이 단계 전체를 별도 서브에이전트로 위임할 필요는 없다(웹앱 Build와 달리 범위가
  작다). 다만 여러 편을 동시에 써야 하는 등 작업이 커지면 Agent 서브에이전트로
  분리해도 된다.

### 3단계: Review (자체 점검)

작성 후 아래를 확인한다(문제가 있으면 즉시 고친다):

- `python3 -m json.tool posts/index.json`으로 JSON 문법 확인
- frontmatter가 `---\n...\n---\n`형식으로 정확히 닫혀 있는지, 다른 글과 필드명이
  일치하는지(`title/date/tags/description`)
- 본문에서 참조하는 링크(예: 앱 링크 `apps/{앱}/index.html`)가 실제로 존재하는지
- 간단한 로컬 서버(`python3 -m http.server`)로 `posts/index.json`과 새 `.md` 파일이
  200으로 응답하는지 curl로 확인 — 브라우저 전체 구동 테스트까지는 보통 불필요하지만
  글에 인터랙티브 요소나 특수 마크다운 문법이 들어가면 실제 브라우저에서도 확인한다

### 4단계: 보고, 커밋, 푸시

- 무엇을 어디에 썼는지 간단히 요약한다.
- **글 게시(post-article 스킬 작업)는 3단계 자체 점검을 통과하면 별도 요청 없이
  바로 git commit + push까지 진행한다.** 2026-07-18 사용자가 명시적으로 이렇게
  지시함 — CLAUDE.md 공통 규칙("승인 없이 구현을 시작하지 않는다")의 커밋 관련
  기본값을 글 게시 워크플로우에 한해 뒤집는 예외다. 웹앱(Plan→Build→Review→Embed)
  등 이 스킬 밖의 다른 작업에는 적용되지 않는다.
  - 커밋 메시지는 간결하게(무엇을 게시했는지), 저장소 기존 커밋 스타일을 따른다.
  - push는 현재 브랜치의 upstream으로 진행한다. upstream이 없거나 origin 자체가
    없는 등 push가 실패하면 사용자에게 알리고 원인을 설명한다(강제 push는 하지 않음).
  - 이 예외는 어디까지나 "정상적으로 검증을 통과한 글 게시"에 한한다. 리뷰에서
    문제가 발견되면 고친 뒤에 커밋하고, 사용자가 이번만 커밋을 보류해달라고
    말하면 그 지시를 따른다.

## 참고

- 예시 포스트 3편의 정확한 형식은 `posts/hello-world.md`, `posts/today-i-learned.md`,
  `posts/markdown-guide.md`를 직접 열어서 확인하는 것이 가장 정확하다(이 스킬 파일에
  내용을 복제하지 않는다 — 원본이 바뀌면 여기 사본이 낡은 정보가 되므로).
- 이 스킬은 계속 업데이트 대상이다. 사용자가 새로운 글쓰기 관례(예: 시리즈물, 다른
  섹션 구성, 커밋 자동화 허용 등)를 알려주면 이 파일의 해당 절을 그때 고친다.

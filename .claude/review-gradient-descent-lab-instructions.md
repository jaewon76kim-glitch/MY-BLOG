Build 에이전트는 다음을 주장했다. **이 주장들을 그대로 믿지 말고 코드를 직접 읽고 재계산해서 검증해라**:
- 릿지를 `J(w) = MSE(w) + (λ/n)‖w‖²`로 MSE-정규화 방식으로 구현했고, 그 결과 λ=0일 때 닫힌 형태 해가
  OLS와 정확히 일치하며 그라디언트도 일관된다고 주장.
- 기본 데이터(9개 점)에서 헤시안 최대 고유값이 약 55.8이라 η<0.036이 안정성 임계값이며, 그래서 기본
  η를 0.02로 낮췄다고 주장 — 이 수치 자체를 직접 재계산해 확인해라.
- 배치/SGD 두 경로를 체크박스로 동시에 표시 가능하게 만들어 spec의 "나란히 오버레이 비교" 요구사항을
  충족했다고 주장.
- Playwright 헤드리스 브라우저로 콘솔 에러 없음, 발산/수렴 동작 확인했다고 주장.

### 4. 수학/로직 재검증 (Node로 regression.js/gradient-descent.js를 직접 로드하거나 로직을 재구현해 계산)

이 항목이 이 리뷰에서 가장 중요하다. 아래를 직접 숫자로 재현할 것 — Build 보고서의 수치를 베끼지 말고
네가 독립적으로 계산해서 일치하는지 대조해라.

- [ ] `MSE(w0,w1) = (1/n)Σ(w0+w1*x_i-y_i)^2`와 그 그라디언트가 코드에 정확히 구현되어 있는가
      (유한차분법으로 수치 미분과 대조해도 좋음: `(MSE(w0+ε,w1)-MSE(w0-ε,w1))/(2ε)` ≈ `∂MSE/∂w0`)
- [ ] 정규방정식 2×2 역행렬 공식이 올바른가. 손으로 만든 간단한 예(예: 데이터 3~4점)에서 OLS 해를
      직접 계산해 코드 결과와 비교
- [ ] **λ=0일 때 릿지 닫힌 형태 해가 OLS 해와 정확히 일치하는가**(부동소수점 오차 범위 내, 1e-9 수준)
      — Build가 주장한 핵심 성질이므로 반드시 직접 재현해서 확인
- [ ] λ를 키울수록 릿지 해의 노름(`‖w‖`)이 작아지는가(원점 방향 이동)
- [ ] 배치 GD를 작은 η로 충분히 반복하면 OLS 해에 수렴하는가(예: η=0.01, 수천 스텝, 상대오차 1% 이내)
- [ ] Build가 주장한 헤시안 최대 고유값(~55.8)과 안정성 임계값(η<0.036)을 기본 데이터(9개 점)에 대해
      직접 재계산해 맞는 수치인지 확인. `Hessian = (2/n) XᵀX` 형태이므로 2×2 행렬의 고유값은 손으로도
      계산 가능(특성방정식 λ²-tr·λ+det=0)
- [ ] 과도하게 큰 η에서 실제로 발산(MSE가 스텝마다 증가)하는지 코드 로직상 확인
- [ ] SGD 스텝이 매번 무작위 데이터 1개만 사용하는지(전체 데이터를 쓰는 배치와 코드가 실제로 다른지)
      확인

### 5. UI/CSS

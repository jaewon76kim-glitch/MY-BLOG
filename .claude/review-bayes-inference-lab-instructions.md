### 4. 수학/로직 재검증 (Node로 bayes-math.js 등을 직접 로드하거나 로직을 재구현해 계산)

이 항목이 이 리뷰에서 가장 중요하다. 아래를 직접 숫자로 재현할 것 — Build 보고서의 수치를 베끼지 말고
네가 독립적으로 계산해서 일치하는지 대조해라.

- [ ] 섹션 1: `p(양성) = p(양성|암)*p(암) + p(양성|암 아님)*(1-p(암))`,
      `p(암|양성) = p(양성|암)*p(암)/p(양성)` 공식이 코드에 정확히 구현되어 있는가. 초깃값(0.8, 0.096,
      0.01)에서 0.008/0.10304 ≈ 0.0777을 직접 재계산해 코드 결과와 대조. 다른 입력값 조합(예: 0.9,
      0.05, 0.02)으로도 직접 계산해 코드가 올바르게 재계산하는지 확인. 100명 격자의 4색 칸 수가
      실제로 `100*p(암)*p(양성|암)`(진짜양성) 등의 공식과 일치하는지 확인
- [ ] 섹션 2: `p(A+|공부함) = 공부함∩A+/공부함전체`, `p(공부함|A+) = 공부함∩A+/A+전체` 공식이 4칸
      입력값을 바꿀 때마다 정확히 재계산되는가. 프리셋1(18/2/2/8→0.9/0.9)과 프리셋2(10/10/2/8→
      0.5/0.8333)을 직접 재현. 임의의 제3의 숫자 조합으로도 직접 계산해 대조
- [ ] 섹션 3: `p(y=c|x) = p(x|y=c)*p(y=c) / Σp(x|y=c')*p(y=c')` 정규화 공식이 코드에 정확히 있는가.
      초깃값(우도 0.75/0.25/0.50, 균등 사전 1/3)에서 A=0.5, B≈0.16667, C≈0.33333을 직접 재계산해
      대조. 사전확률을 비균등(예: 0.5/0.3/0.2)으로 바꿔도 정규화가 정확한지 확인
- [ ] 섹션 4: `p(D|h) = (1/|h|)^N`(D의 모든 원소가 h에 속할 때만, 아니면 모순=0) 공식이 코드에
      정확히 있는가. 가설공간 11개 각각의 `|h|` 값을 1~100 범위에서 직접 세어(소수 25개, 완전제곱수
      10개, 3의 배수 33개 등) 코드의 값과 일치하는지 확인. `D={16}`→0.16667/0.02,
      `D={16,8,2,64}`→우도비 약 4822.53:1을 직접 재계산해 대조. h_two가 "1을 제외한 2의 거듭제곱
      {2,4,8,16,32,64}, |h|=6"으로 정의되어 있는지(원문의 "6개" 표현과 일치시키기 위한 의도적 선택)
      확인
- [ ] 섹션 5: h' = "32 제외 2의 거듭제곱"(|h'|=5)이 코드에 정의되어 있는가. N=4일 때
      `p(D|h')=(1/5)^4=0.0016`이 `p(D|h_two)=(1/6)^4≈0.00077`보다 커서 우도만으로는(MLE) h'가 이기고,
      사전확률을 반영하면(MAP) h_two가 이기는지 직접 재계산해 확인. N을 늘려가며(예: N=10, N=25, N=40)
      사전 항의 상대 비중이 실제로 줄어드는지, "N≈25.3에서 교차한다"는 주장이 실제 코드/공식으로
      성립하는지 직접 계산으로 검증(사전확률 값 자체가 무엇으로 설정되어 있는지 코드에서 확인 후 재현)
- [ ] "놀라움(surprisal)" 표시 방식 변경이 실제로 부호만 뒤집은 것이고 지침의 공식
      (`log p(D|h) = -N*log|h|`, `log p(h)`는 N과 무관한 상수)과 수학적으로 동치인지 코드를 읽고 확인
